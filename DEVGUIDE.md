# 개발 원칙 (Development Guidelines)

모든 기능 개발 시 아래 원칙을 준수한다. AI 에이전트 및 개발자 모두 이 문서를 최우선 참고 기준으로 삼는다.

---

## 1. 프론트엔드 렌더링 최적화
- 불필요한 리렌더링 최소화. 상태(state)는 필요한 최소 범위에만 선언한다.
- 서버 컴포넌트(Server Component)를 기본으로 사용하고, 클라이언트 컴포넌트(`"use client"`)는 인터랙션이 반드시 필요한 경우에만 적용한다.
- 리스트 렌더링 시 안정적인 `key`를 사용한다. 인덱스 key 사용 금지.
- `memo`, `useMemo`, `useCallback`은 실제 성능 문제가 측정된 경우에만 사용한다. 예방적 최적화 금지.

## 2. DB 최적화
- 필요한 컬럼만 SELECT한다. `SELECT *` 금지.
- N+1 쿼리 금지. 연관 데이터는 JOIN 또는 일괄 조회로 처리한다.
- 자주 조회되는 컬럼에 인덱스를 반드시 설정한다.
- 단순 조건 필터링은 애플리케이션 레이어가 아닌 DB 레이어에서 처리한다.

## 3. API 호출 최적화
- 한 화면에서 필요한 데이터는 `Promise.all`로 병렬 호출한다. 순차 호출 금지.
- 동일한 데이터를 여러 곳에서 중복 호출하지 않는다. 상위에서 한 번 조회 후 props로 전달한다.
- 클라이언트 측 fetch는 SWR 또는 React Query 등 캐시 레이어를 통해 관리한다.
- 페이지네이션, 무한스크롤 등 대량 데이터는 반드시 분할 조회한다.

## 4. 이미지 최적화
- 모든 이미지는 **WebP** 포맷을 기본으로 사용한다.
- 썸네일 기준 사이즈: **640×360px** (16:9, Retina 2x 기준).
- 뷰포트 밖 이미지에는 `loading="lazy"`를 적용한다. Above-the-fold 이미지는 `loading="eager"` + `fetchpriority="high"`.
- 이미지 `width`, `height` 속성을 명시해 CLS(Cumulative Layout Shift)를 방지한다.
- Next.js `<Image>` 컴포넌트를 사용할 수 있는 경우 `<img>` 대신 사용한다.

## 5. 캐시 활용
- 변경이 드문 데이터(템플릿 목록, 설정값 등)는 서버 캐시 또는 `revalidate`를 적용한다.
- 동일 요청 내에서 이미 조회한 데이터를 재조회하지 않는다. 변수로 재사용한다.
- 클라이언트 전역 상태(Zustand, Context 등)에 서버 데이터를 캐시할 때는 stale 여부를 반드시 관리한다.

## 6. 개발 코드와 UI 코드 분리
- 비즈니스 로직(`store`, `actions`, `utils`)과 UI 컴포넌트를 같은 파일에 혼재 금지.
- 서버 액션은 `src/app/*/\_actions/` 디렉토리에 위치한다.
- 스토어/데이터 레이어는 `src/lib/` 디렉토리에 위치한다.
- UI 컴포넌트는 `src/components/` 디렉토리에 위치한다. 페이지 전용 컴포넌트는 해당 라우트 하위에 위치한다.

## 7. 파일 분리 원칙
- 한 파일은 하나의 책임만 갖는다 (Single Responsibility).
- 파일 300줄 초과 시 분리를 검토한다.
- 타입 정의는 `types.ts`에 집중 관리한다.
- 유틸리티 함수는 기능별로 `utils/` 하위에 분리한다.
- 관련 없는 기능을 하나의 파일에 묶지 않는다.

## 8. 모니터링 및 로그 수집
- 에러는 반드시 try-catch로 처리하고 적절한 로그를 남긴다. 에러를 조용히 삼키는 코드 금지.
- 브라우저: Core Web Vitals(LCP, CLS, FID) 측정 및 수집 체계를 갖춘다.
- 서버: API 응답 시간, 오류율을 추적한다.
- DB: 슬로우 쿼리(500ms 이상)를 감지하고 로그로 기록한다.
- 프로덕션 배포 전 에러 로그 수집 체계(Sentry 등)를 연동한다.

## 9. 코드 품질 원칙
- 중복 코드 금지. 3회 이상 반복되는 로직은 함수/컴포넌트로 추출한다.
- 불필요한 추상화 금지. 실제로 재사용되지 않는 코드를 미리 일반화하지 않는다.
- 과한 연산(루프 내 중첩 루프, 불필요한 정렬 등)은 사전에 데이터 구조로 해결한다.
- 사용되지 않는 변수, import, 함수는 즉시 제거한다.
- `any` 타입 사용 금지. 타입을 명확히 정의한다.

## 10. 단계별 리팩토링 원칙
- 기능 단위(Feature) 또는 그룹 단위 작업 완료 후, **다음 작업 착수 전에** 리팩토링 필요 여부를 먼저 제안한다.
- 리팩토링 제안 항목: 중복 코드, 파일 분리, 타입 개선, 성능 개선, 테스트 추가.
- 리팩토링은 기능 변경 없이 구조만 개선한다. 기능 추가와 리팩토링을 동시에 진행하지 않는다.
- 사용자가 리팩토링을 승인한 경우에만 진행한다.
