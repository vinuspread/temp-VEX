# PRD: 템플릿허브 운영 요구사항 (현재 운영 기준)

## 1. 목적
- 템플릿허브에서 템플릿을 **등록/매핑/노출/신청관리**할 수 있는 운영 기준을 정의한다.
- 실행 템플릿 소스는 GitHub 코드 기준으로 관리하고, 관리자에서는 메타데이터를 관리한다.

## 2. 핵심 원칙
1. 템플릿 실행은 템플릿허브 내부 라우트에서만 수행한다.
2. 관리자 등록은 업로드가 아니라 **GitHub 템플릿 카탈로그 매핑** 방식으로 처리한다.
3. UI 요소와 비즈니스 로직은 분리한다.
4. 관리자 UI는 `admin-designsystem.json` 기준을 따른다.

## 3. 관리자 기능

### 3-1. 인증
- Google OAuth 로그인
- `ADMIN_ALLOWLIST_EMAILS` 기반 관리자 허용

### 3-2. 템플릿 등록/수정
- 필수 입력
  - 템플릿명
  - 카테고리
  - JS 유형(분류값)
  - 노출 상태
  - 신청 가능 여부
  - 서브 페이지 수
  - 실행 템플릿 라우트(카탈로그 선택)
  - 한 줄 요약
  - 상세 설명
  - 기본 썸네일 3장 이상(드래그앤드롭)
- 선택 입력
  - 태그
  - 버전/개정 메모
- `slug`는 자동 생성(소문자 + 난수 5자리), 수동 입력 없음
- 동일 실행 라우트 중복 매핑 금지

### 3-3. 카테고리 관리
- 카테고리 추가/수정/비활성화
- 중복 이름 금지

### 3-4. 신청 관리
- 신청 목록/상세 조회
- 상태 변경 (`RECEIVED`, `REVIEWING`, `DONE`, `ON_HOLD`)
- 관리자 메모/처리 이력 관리

### 3-5. 백업/복구
- 관리자 수동 백업 버튼 제공
- 정기 백업(6시간 단위)
- 백업 파일 4개 유지(신규 생성 시 가장 오래된 백업 삭제)
- 복구 시 즉시 적용 및 운영 화면 검증

## 4. 사용자 기능
- 공개 템플릿 목록 조회
- 템플릿 미리보기(선택 라우트)
- 템플릿 신청(신청 불가/비공개 정책 반영)

## 5. GitHub 템플릿 카탈로그
- 경로: `template-sources/catalog.json`
- 항목: `key`, `name`, `runtimeRoute`, `sourcePath`
- 템플릿 등록 시 카탈로그 항목을 선택해 실행 라우트를 매핑한다.

## 6. 데이터 저장
- 템플릿/카테고리/신청/백업 상태는 Supabase `admin_state`에 저장한다.
- 실행 소스코드 자체는 GitHub 저장소 코드가 기준이다.
